# OODA

使用本 skill 运行强制 OODA 执行循环：澄清目的，观察足够的现实，围绕真实问题定向，明确决策，在范围内执行，同时验证结果和路径；未解决就自动下一轮，有价值时沉淀学习，只有在问题被验证解决后才关闭。

这是中文版。默认英文版位于 `SKILL.md`。

面向用户的输出默认使用用户所用语言，除非用户另有要求。

## 首要动作：状态栏任务初始化

**Skill 激活后第一个动作：创建第 1 个任务。不先读文件、不先分析。**

### 核心规则：一次只创建一个任务

**禁止一次性创建全部 5 个任务。** 每完成一个任务才创建下一个。状态栏始终保持 2-3 个任务，天然有序。

```
创建"1/5 理解问题" → in_progress → completed
然后创建"2/5 对齐提问" → in_progress → completed
然后创建"3/5 诊断定向 — R1" → in_progress → completed
然后创建"4/5 决策执行 — R1" → in_progress → completed
然后创建"5/5 验证关闭 — R1" → in_progress → completed
[如需新一轮]
然后创建"3/5 诊断定向 — R2" → in_progress → completed
然后创建"4/5 决策执行 — R2" → in_progress → completed
然后创建"5/5 验证关闭 — R2" → in_progress → completed
```

### 5 个任务（逐个创建）

```
TaskCreate: "1/5 理解问题 — 澄清目的·定义成功标准"
TaskCreate: "2/5 对齐提问 — 复述理解·收集约束"
TaskCreate: "3/5 诊断定向 — R1 观察现状·分析根因"
TaskCreate: "4/5 决策执行 — R1 选择方案·落地变更"
TaskCreate: "5/5 验证关闭 — R1 检查结果·总结迭代"
```

### 推进规则（单轮 5 任务 + 多轮循环）

```
前置（只做一次）:
  [创建] 1/5 理解问题 → in_progress → completed
  [创建] 2/5 对齐提问 → in_progress → completed

循环（每轮重建 3/4/5，带轮次标记 R1/R2/...）:
  [创建] 3/5 诊断定向 — R1 观察现状·分析根因 → in_progress → completed
  [创建] 4/5 决策执行 — R1 选择方案·落地变更 → in_progress → completed
  [创建] 5/5 验证关闭 — R1 检查结果·总结迭代 → in_progress → completed

如果验证未通过 → 自动下一轮：
  [创建] 3/5 诊断定向 — R2 观察现状·分析根因 → in_progress → completed
  [创建] 4/5 决策执行 — R2 选择方案·落地变更 → in_progress → completed
  [创建] 5/5 验证关闭 — R2 检查结果·总结迭代 → in_progress → completed
```

| 任务 | 生命周期 | 创建时机 | completed 时机 |
|------|---------|---------|---------------|
| 1/5 理解问题 | 仅一次 | 激活后立刻 | Purpose 写完后 |
| 2/5 对齐提问 | 仅一次 | 任务 1 completed 后 | U&Q 回答后 |
| 3/5 诊断定向 — R{N} | 每轮重建 | 任务 2 completed（首轮）/ 上轮结束（后续轮） | Orient 输出后 |
| 4/5 决策执行 — R{N} | 每轮重建 | 任务 3 completed 后 | Act 完成后 |
| 5/5 验证关闭 — R{N} | 每轮重建 | 任务 4 completed 后 | 验证通过并 Close 后 |

**每一行强制。不得合并、不得跳过、不得一口气创建。一轮结束后自动进入下一轮直到问题验证解决。** 

### 用户确认必须用 AskUserQuestion

以下三个检查点**必须**使用 `AskUserQuestion` 弹出提交框让用户确认，**禁止**只用纯文本等回复：

| 检查点 | 问题示例 |
|--------|---------|
| 理解回执 | "我的理解对吗？" → 选项：确认 / 需要调整 |
| 深入提问 | 列出 3-4 个核心问题，每个给 A/B/C 选项 |
| Decide 决策 | "以下方案可以执行吗？" → 选项：确认执行 / 换方案 / 补充信息 |

`AskUserQuestion` 最多 4 个问题，每个最多 4 个选项。U&Q 阶段如果问题超过 4 个，分两批问。 

新一轮时：创建新的"第N轮·诊断定向/决策执行/验证关闭"任务（任务 3-5），流程同上。

**琐碎任务**（单行修复、改一个字）→ 不用 TaskCreate，直接执行。

## 目的优先

第一轮开始前，先判断这个任务是否需要 OODA：

- **琐碎任务**（单行修复、改一个词、typo）→ 直接执行，不需要 OODA。告知用户跳过了循环及原因。
- **简单但可验证**（单文件修改、范围明确）→ 微循环。
- **复杂或模糊**（多文件、范围不清、需要用户偏好）→ 中循环及以上，走完整 Purpose First。

如果任务达到门槛，定义：

- 用户的真实目的。
- 本任务的成功标准。
- 已知约束、风险和必须确认的事项。

如果其中任何一项不清楚，向用户提问，不要自行推断。记录状态为 `paused_waiting_user_input`，用户回答后继续同一个 OODA 任务。

如果任务通过了门槛（非琐碎），立即进入下方的"理解回执"，然后再进入"理解与提问"，最后才开始第一轮 OODA。

## 理解回执

在 Purpose First 之后、U&Q 之前，用自己的话向用户重述理解。**这一步在方向层面的猜测落地之前先让用户校验。**

### 规则

1. **用一段话重述。** 覆盖四个维度：
   - **要做什么**（核心目标，一句话）
   - **为什么做**（任务背后的上下文或问题）
   - **边界在哪**（范围内 vs. 明确不在范围内）
   - **怎么做**（高层策略，不是详细步骤）

2. **暴露你的假设。** 如果理解的任何部分依赖推断而非用户的明确输入，标注出来："我假设 [X]，基于 [Y]。对吗？"

3. **用 AskUserQuestion 确认。** 以 `AskUserQuestion` 弹出提交框，给用户"确认"或"需要调整"两个选项。**禁止只用纯文本等回复。**

4. **等待确认。** 用户确认或纠正后才进入 U&Q。如果用户纠正，重述修正后的理解，再次确认。

5. **写入文件。** 将重述内容和用户反馈（确认或纠正了什么）记录到 `00-purpose.md` 的 `## 理解回执` 章节。

### 输出格式

```text
## 理解回执
- 要做什么：[一句话 — 核心目标]
- 为什么做：[一句话 — 上下文或问题]
- 边界：范围内 → [X]；范围外 → [Y]
- 怎么做：[高层策略，1-2 句]
- 暴露的假设：[列出标记为待确认的推断]
- 用户反馈：已确认 | 已纠正 → [改了什么]
```

这一步不替代 U&Q——它先对齐方向。U&Q 再处理细节层面的未知（范围边界细节、深度、规范、依赖、偏好）。

## 理解与提问

在 Purpose First 之后、进入第一轮 OODA 之前，必须执行"理解与提问"阶段。

### 规则

1. **先读上下文。** 读取 CLAUDE.md、相关项目文件、目录结构、最近变更等，理解任务环境。

2. **强制校验——即使用户给了方案也必须提问。** U&Q 不是"有不懂的才问"，而是"不问就不许进入第一轮"。用户给了明确的方案 ≠ 你可以跳过 U&Q。你必须从以下四个维度各提至少 1 个校验问题（共至少 3 个；分析/评估/诊断类任务至少 4 个），不能以"用户已经说清楚了"为理由跳过：
   - **边界校验**：方案覆盖了什么、没覆盖什么？边界之外有什么风险？
   - **风险校验**：方案中最可能出错的环节是什么？有没有隐藏假设？
   - **遗漏校验**：方案中没提到但可能需要的依赖、步骤或回退计划是什么？
   - **对标校验**（分析/评估/诊断类任务强制执行；编码/修复/文档类任务跳过）：必须逐条问清以下信息，不得合并为一个话题标签——
     · 是否存在竞品、对标对手、参考项目或行业基准？（无竞品也是有效答案，但必须明确确认）
     · 如果有：对方当前状态、关键指标、核心能力是什么？
     · 用户在对标中处于什么位置？（领先/追赶/持平/不相关）
     · 有没有可参照的行业标准或通行做法？

3. **补充发现式提问。** 在强制校验之外，找出所有你不理解或不确定的地方：
   - 范围边界（到底哪些在范围内、哪些不在？）
   - 深度预期（表面修还是深重构？）
   - 项目约定（命名、结构、已有模式？）
   - 依赖关系（目标依赖于什么？什么依赖于目标？）
   - 用户偏好（多个可行方案中，用户倾向哪个？）
   - 用户请求中任何模糊的术语、概念或引用。

4. **用 AskUserQuestion 提问。** 把问题（强制校验 + 发现式提问）通过 `AskUserQuestion` 弹出提交框。每个问题给 A/B/C 选项。最多 4 个问题，超过则分两批。每个问题必须：
   - **具体且闭合**：能用选择回答。
   - **关于真正的未知**：不是读文件就能查到的。
   - **按优先级排序**：影响力最大的排前面。

5. **等待全部回答。** 在用户回答完所有问题之前，不得进入下一步。如果用户只回答了部分，追问未回答的部分。

6. **总结并确认。** 收到所有回答后，用 2-3 句话总结你的理解，问用户"这符合你的预期吗？"得到明确确认后再进入 Observe。

7. **写入文件。** 把问题、回答和确认后的理解写入 `00-目的.md` 的 `## 理解与提问` 部分。

### 输出格式

```text
## 理解与提问
- 已读上下文: [检查了哪些文件和目录]
- 发现的未知项: [列出了哪些不清楚的地方]
- 提出的问题:
  1. [问题1] → 回答: [用户的回答]
  2. [问题2] → 回答: [用户的回答]
- 确认后的理解: [2-3 句话总结]
- 用户已确认: 是
```

### 重新触发

如果在后续 OODA 轮次中出现了新的、事先没有预料到的、读文件也无法解决的未知项，暂停当前轮次，重新进入"理解与提问"阶段。用户回答后，从 Observe 重新开始当前轮次——新信息可能改变你之前看到的东西。永远不要猜。

## 过程记录目录

OODA 过程文件写入 `work_ooda/`。该目录是 OODA 的工作台，用于记录思考、执行和复盘，**不是任务最终交付物目录**。

在**当前工作目录**下创建，不要创建在 skill 安装目录下：

```text
work_ooda/{YYYYMMDD-HHMMSS}-{任务名}/
```

`YYYYMMDD-HHMMSS` 是任务开始时的本地时间戳。`{任务名}` 从用户请求中提取。

使用以下文件结构：

```text
work_ooda/
├── _index.md          ← 所有任务的摘要索引，每次 Close 追加一行
├── {YYYYMMDD-HHMMSS}-{任务名}/
│   ├── 00-目的.md
│   ├── 01-第一轮.md
│   ├── ...
│   └── summary.md
```

每轮结束后立即写入单独文件。不要只把所有轮次写进一个 summary。

**不可变规则**：轮次文件写入后不得修改。如果后续轮次发现之前的观察或判断有误，不要编辑旧文件。在新轮次的 Observe 中标注：`[修正第N轮] 原判断: X。修正: Y。原因: Z。`

重要：`work_ooda/` 只存放 OODA 元文件：目的澄清、每轮报告、最终总结。任务实际交付物，例如代码、文档、报告或分析成果，必须写入项目约定位置，**禁止放入 `work_ooda/`**。

`00-目的.md` 必须包含：

- 用户目标：用户真正想要的效果，而不仅是任务字面表述。
- 成功标准：如何判断任务完成。
- 约束：时间、范围、安全边界、必须确认的事项。
- 理解回执：要做什么/为什么做/边界/怎么做、暴露的假设、用户反馈（已确认或已纠正）。
- U&Q：提出的问题、收到的回答、确认后的理解。
- 预计轮次：能估计就写数字，不能估计写 `unknown`。

每个 `0{N}-第{N}轮.md` 必须包含：

- Observe 证据。
- Orient 诊断与问题分级。
- Decide 方案。
- Act 变更。
- 自检。
- 是否进入下一轮。

`summary.md` 必须包含：

- 目的回顾。
- 总轮次和最终状态：`完成`、`部分完成` 或 `需要人工`。
- 每轮摘要表。
- 成功标准验收清单。
- 仍需用户决策的问题。

记录保持简洁。不要存储密钥、凭证、私有 token 或不必要的用户数据。

## 详细输出模式

每次 OODA 都必须使用详细输出模式。以结构化方式向用户展示每一步的当前逻辑。

### 每步输出格式

| 步骤 | 输出内容 | 任务操作 |
| --- | --- | --- |
| Purpose | `## 目的澄清` + 用户目标、成功标准、约束条件 | 完成后 → 任务 1 `completed`；任务 2 → `in_progress` |
| 理解回执 | `## 理解回执` + 要做什么/为什么做/边界/怎么做 + 暴露的假设 + 用户反馈 | 用户确认后 → 任务 2 `completed`；任务 3 → `in_progress` |
| Understand & Question | `## 理解与提问` + 已读上下文、未知项、问题、回答、确认后的理解 | U&Q 全部回答后 → 任务 3 `completed`；任务 4 → `in_progress` |
| Observe | `## Observe 观察` + 正在观察什么、当前状态、关键发现；第 2 轮起追加 Delta 对比 | — |
| Orient | `## Orient 定向` + 一句话根因和问题分级表 | Orient 输出后 → 任务 4 `completed`；任务 5 → `in_progress` |
| Decide | `## Decide 决策` + 选择方案、选择原因、预期效果 | — |
| Act | `## Act 执行` + 具体操作、涉及文件、执行结果和错误信息 | Act 完成后 → 任务 4/5 `completed`；任务 5/5 → `in_progress` |
| Verify | `## Verify 验证` + 验证方式、验证结果、置信度和回归状态 | — |
| Iterate/Close | `## 迭代 / 关闭` + 下一轮 或 关闭总结 | 下一轮 → 创建新任务 3/4/5；关闭 → 任务 5/5 `completed` |

同时写入过程文件，并向用户展示步骤摘要。

### 进度指示器

进度通过 Claude Code 原生任务系统在状态栏显示，5 个任务纵向排列，多轮时 R1/R2 标记清晰：

```
1/5 理解问题 — 澄清目的·定义成功标准
2/5 对齐提问 — 复述理解·收集约束
3/5 诊断定向 — R1 观察现状·分析根因      ← 轮次标记
4/5 决策执行 — R1 选择方案·落地变更
5/5 验证关闭 — R1 检查结果·总结迭代
[如有 R2] 3/5 诊断定向 — R2...  ← 新一轮任务叠加显示
```

每个任务在对应阶段通过 `TaskUpdate` 设为 `in_progress` (●) 或 `completed` (◼)。

**步骤缩写对照**（仅用于聊天内的文本标注）：

| 缩写 | 步骤 | 时机 |
|------|------|------|
| P | Purpose First（目的澄清） | 前置，仅一次 |
| U | Understanding Restatement（理解回执） | 前置，仅一次 |
| Q | Understand & Question（理解与提问） | 前置，仅一次 |
| O | Observe（观察） | 每轮 |
| R | Orient（定向） | 每轮 |
| D | Decide（决策） | 每轮 |
| A | Act（执行） | 每轮 |
| V | Verify（验证） | 每轮 |
| I | Iterate（迭代，进入下一轮） | 每轮末尾（如还有下一轮） |
| C | Close（关闭） | 最后一轮（替代 I） |

**显示规则**：
- 任务 1-2（理解问题/对齐提问）依次推进，完成后置为 completed
- 任务 3-5 带轮次标记（R1、R2...），完成后保留，当前进度一目了然
- 进入新一轮 → 创建新的 3/4/5（R{N+1}），旧轮次保持 completed
- 状态栏显示：◼ completed（绿） / ● in_progress（黄） / ◻ pending

## 循环规模（白话：这轮要花多少力气。规模越大，证据要越多、验证要越严。）

按两个独立维度选择循环规模：
- **复杂度**：范围多大？单文件 → 多文件 → 跨系统。
- **不可逆程度**：回退多难？可逆 → 需确认 → 不可逆。

复杂度越高或不可逆程度越高 → 规模越大。

| 规模 | 复杂度 | 不可逆程度 | 使用场景 | 典型形态 |
| --- | --- | --- | --- | --- |
| 微循环 | 单文件 | 可逆 | 窄范围 bug、文档修复、单文件修改。对比或验证类任务如果手动检查可能发现结构性问题，不要用微循环——用中循环 | 观察相关对象，做一个范围明确的决策，执行，然后再跑一轮验证。涉及文件对比、结构阅读或手动逻辑走查的任务默认走中循环 |
| 中循环 | 多文件 | 需确认 | 功能、多文件 skill 更新、工作流设计或非平凡内容包 | 按依赖拆分，每轮处理选定问题，用测试或结构化检查验证 |
| 宏循环 | 跨系统 | 不可逆 | 架构、迁移、发布或高风险策略 | 尽早询问成功标准，使用可见检查点，不可逆动作前暂停确认 |

不要因为有规模分类就把普通工作变慢。规模只决定每轮需要多少证据、规划和验证。

## 最小输出契约 — 每个完成任务结尾必须包含的最基本信息

已完成的工作，结尾必须包含：

- 具体结果。
- 执行过的验证。
- 问题是否已解决。
- 过程记录目录：`work_ooda/{YYYYMMDD-HHMMSS}-{任务名}/`，仅包含 OODA 元文件，不是任务交付物。
- 总轮次。
- 只有当仍有下一步时才写下一步。

决策类任务必须说明：

- 选择的路径。
- 为什么该路径符合当前证据。
- 是否需要用户确认后再行动。

## 循环

### 详细输出总则

每一步向用户输出当前逻辑。使用 `## 步骤名` 作为标题，1-3 句话具体逻辑。不输出空标题、空话或冗长仪式文字。展示具体对象、具体问题和具体决策。

### 0. Purpose（目的）

进入循环前先澄清用户目的。

输出：

```text
## 目的澄清
- 用户目标: [一句话]
- 成功标准: [列表]
- 约束条件: [列表]
- 预计轮次: [数字或 unknown]
- 状态: ready | paused_waiting_user_input
```

同时写入 `00-目的.md`。如果目的或成功标准不清楚，向用户提问并暂停。

目的确认后 → `TaskUpdate` 任务 1 `completed`；任务 2 → `in_progress`。

### -1. 理解与提问（第一轮前强制执行）

遵循上方[理解与提问](#理解与提问)章节的规则和输出模板。将 Q&A 写入 `00-目的.md`。

U&Q 全部回答后 → `TaskUpdate` 任务 3 `completed`；任务 4 → `in_progress`。

### 每轮循环

Orient 输出后 → `TaskUpdate` 任务 4 `completed`；任务 5 → `in_progress`。

Act 完成后 → `TaskUpdate` 任务 4/5 `completed`；任务 5/5 → `in_progress`。

验证通过后：
- 进入下一轮 → `TaskUpdate` 任务 5/5 `completed`；创建新任务 `3/5 诊断定向 — R{N}` → `in_progress`，然后依次 4/5、5/5
- 关闭 → 任务 5/5 `completed`，进入 Close

### 1. Observe（观察）

捕获当前状态，聚焦关键信息。

必须实际读取本轮涉及的文件或对象。不要依赖记忆。

输出：

```text
## Observe 观察
- 当前状态: [描述当前实际状态]
- 关键发现（每条标注来源类型）:
  - [直接观察] 纯感官事实——只描述实际看到、读到、返回了什么。禁止使用任何规范性词汇：不得出现"应该""缺少""正确""错误""规范""合理""正常""异常""有问题"（作为判断时）。如果描述中隐含了判断标准，就必须标为 [AI推断]。
  - [AI推断] 从观察到的事实中做的推理——仍是假设，可能出错
  - [AI结论] 综合多项观察和推断后得出的判断

  规则：每条关键发现必须标注来源类型。禁止把推断写成观察、把结论写成事实。

  正反例对照——以下哪些算 [直接观察]：
  ❌ "第42行缺少空指针检查" → [AI推断]（"缺少"隐含了"应该有空指针检查"的标准）
  ✅ "第42行代码为 obj.method()，其上方没有 null 检查语句" → [直接观察]
  ❌ "配置文件格式不规范" → [AI推断]（"规范"是一个判断标准）
  ✅ "配置文件缩进为 3 空格，项目其余文件缩进为 2 空格" → [直接观察]
  ❌ "API 响应时间过长" → [AI推断]（"过长"隐含了性能基线）
  ✅ "API /orders 接口 P99 延迟 = 3500ms" → [直接观察]
```

从第 2 轮起，追加 Delta 对比（白话：拿本轮状态跟上轮比——修好了几个、没生效几个、新出来几个）：

```text
- Delta 对比:
  修复项: X -> 验证通过: Y，未生效: Z
  新引入问题: N
  遗留问题: M
```

只检查真正相关的材料：项目笔记、相关文件、最近变更、用户约束和真实目标。

关键发现之后，做一轮结构感知扫描。三条至少回答一条；确实没有发现时写"无异常"，但不能是唯一内容：

- 结构感知: 这里的隐含结构是什么？核心和边缘分别在哪？
- 矛盾感知: 有什么不应该同时存在但确实同时存在的东西？
- 缺席感知: 什么应该存在但不存在？

### 2. Orient（定向）

快速但有证据地形成判断。

输出：

```text
## Orient 定向
- 根因分析（至少钻两层）:
  - 表象 (What): [可见的问题是什么]
  - 机理 (Why): [直接原因是什么 — 底线层，必须到达]
  - 原理 (Why allowed): [为什么系统允许它发生 — 强烈推荐]

  如果未到达原理层，必须说明具体原因（如"系统原因涉及本任务范围外的 CI 配置"）。
  "当前修复已足够"等通用理由不可接受。
- 与目标的差距: [还差什么才达到成功标准]
- 问题分级:
  | 级别 | 问题 | 标签 |
  | --- | --- | --- |
  | critical/major/minor/cosmetic | ... | ... |
```

识别核心问题、根因、与目标的差距，以及有意义时的 2-3 条可行路径。倾向具体判断。证据不足就直接说明。

问题分级（白话：这个问题有多要命——致命的/重要的/小毛病的/表面的）：

| 级别 | 含义 | 动作 |
| --- | --- | --- |
| `critical` | 阻碍目标达成 | 本轮必须修复 |
| `major` | 影响质量但不完全阻塞 | 本轮优先修复 |
| `minor` | 小瑕疵 | 成本低就修，否则记录到下一轮 |
| `cosmetic` | 格式或命名优化 | 延后或跳过 |

用问题级别决定本轮范围。

### 3. Decide（决策）

选择下一步行动。

输出：

```text
## Decide 决策
- 选择方案: [本轮要做什么]
- 选择原因: [为什么适合]
- 预期效果: [变更后的预期状态]
- 本轮范围: [只处理本轮决定的问题，按 critical -> major -> minor -> cosmetic 优先级]
```

普通低风险工作直接执行。高风险、破坏性、昂贵、外部或模糊操作，必须暂停并请求用户确认。

无独立 TaskUpdate——Decide 在 Orient（任务4）之后、Act（任务5）之前。Orient 输出时任务 4 已完成、任务 5 已 in_progress。若 Decide 阶段需要用户确认，任务 5 保持 in_progress 等待。

### 4. Act（执行）

执行选定步骤。

输出：

```text
## Act 执行
- 具体操作: [做了什么，包括命令或编辑]
- 涉及文件: [修改了哪些文件]
- 执行结果: [成功/失败；失败时写错误和处理方式]
```

执行要求：

- 使用具体命令、文件和证据。避免“可能”“应该没问题”等模糊说法。
- 锁定范围（白话：本轮只修 Decide 选定的问题，不顺手改别的。新发现记入下一轮。）：只修 Decide 选定的问题。保留既有风格和结构。不要混入无关清理或重构。发现新问题记录到下一轮。
- 独立问题可并行，依赖问题串行。
- 明显错误重试一次。否则换方案或向用户求助。

### 5. Verify（验证）

检查行动是否生效。

输出：

```text
## Verify 验证
- 验证方式: [测试、文件检查、目标对比等]
- 验证结果: [问题是否解决，并给出具体证据]
- 剩余问题: [如有，列出]
- 是否引入回归: yes | no；如有标注 [regression]
- 置信度: high | medium | low
```

自检清单，写入每轮文件：

```text
[ ] 修复已生效
[ ] 无新回归，或已列出回归
[ ] 置信度: high | medium | low
[ ] 任务状态栏：5 个任务均按阶段正确 in_progress/completed
[ ] 需要用户提问: yes | no
[ ] 需要下一轮: yes | no
[ ] Guardrails 合规: 逐条检查 5 条 guardrail，每条附带具体证据
```

Guardrails 合规项的写法示例：

```text
[ ] Guardrails 合规:
    - "不猜测" → 在 U&Q 阶段向用户确认了范围边界（见 00-purpose.md）
    - "不跑偏" → 只改了 Decide 列出的 3 个文件，1 个新发现记入下一轮
    - "不跳过" → 重新读取了 config.yaml 再 Orient，构建验证 0 错误
    - "说人话" → 引用了 L42、L105，附带了修改前后片段
    - "留后路" → 在轮次文件中记录了"空指针前检查"模式；已索引到 _index.md（bug-fix/critical）
```

没有证据的 guardrail 勾选框不算合规。

验证通过后：任务 5/5 → `completed`（或进入新一轮）。

验证通过前，必须同时确认：

- 结果满足用户说明的目的和成功标准。
- 执行路径没有跳过必要观察、猜测不清楚的意图，或超出已决定范围。

代码、配置、skill、prompt、模板或 agent 配置类任务，还要检查：测试或校验通过、风格一致、无未使用导入或引用、没有添加未请求功能。

### 6. Iterate（迭代）

验证发现未解决问题时，自动进入下一轮 OODA。

输出：

```text
## Iterate 迭代
- 下一轮: yes | no | paused_waiting_user_input
- 原因: [为什么需要或不需要下一轮]
```

- 未解决问题 -> 自动下一轮，并记录 `next_round: yes`。进入新一轮时创建新的"第N轮"任务（3/5→4/5→5/5）。
- 受阻 -> 记录 `paused_waiting_user_input`，并提出一个简洁问题。
- 修复无效或恶化 -> 标注 `[regression]`，执行最小安全回滚，并换方案。
- 同一问题连续两轮修复无效 -> 暂停求助，记录尝试方案、回滚状态和失败原因。

当某一轮产生了可复用模式、检查清单、命令、测试用例、诊断方法、决策规则或其他资产时，必须记录可复用学习。不要强迫琐碎任务沉淀学习。

有价值时，记录：

- 可复用学习：下次还能使用的判断、经验或规律。
- 可复用资产：本轮产生或改进的测试用例、检查清单、命令、模板、文档、脚本或决策规则。
- 下次捷径：这一轮让下次哪里更快或更稳。

只有在有长期价值时，才把可复用经验写入记忆。

### 7. Close（关闭）

只有问题被验证解决，或用户明确要求停止，才能关闭。

启动关闭：
- 任务 5/5 已在 Verify 阶段设为 `in_progress`，写 summary 时保持
- 全部完成后：`TaskUpdate` 任务 5/5 → `completed`

输出：

```text
## 任务关闭
- 最终结果: [做了什么变更]
- 验证: [验证了什么]
- 为什么已解决: [一句话]
- 过程记录: work_ooda/{YYYYMMDD-HHMMSS}-{任务名}/
- 任务交付物: [实际输出位置，非 work_ooda/]
- 总轮次: N
- 遗留: [如有]
```

关闭前写入 `summary.md`，并逐条检查 `00-目的.md` 中的成功标准。

向 `work_ooda/_index.md` 追加一行：

```text
| YYYY-MM-DD | {任务名} | N轮 | {关键发现一句话} | {任务类型} | {最高级别} | {标签} |
```

- `任务类型`：`bug-fix` | `refactor` | `review` | `analysis` | `content` | `skill-fix`
- `最高级别`：`critical` | `major` | `minor` | `cosmetic`（本轮处理过的最高问题级别）
- `标签`：自由关键词，用于跨任务搜索，如 `#空指针 #认证 #性能`。用 `#` 前缀、空格分隔。至少标注技术领域和症状类型。

关闭完成后：
- `TaskUpdate`: 任务 5/5 `completed`（确认 summary 已写入、_index.md 已更新）

不要仅仅因为执行了某个动作就关闭任务。只有结果和执行路径都已按用户目的验证通过，才能关闭。

## 停止与暂停规则

持续运行 OODA，直到问题被验证解决。

只有以下情况可以关闭：

- 问题已解决，并且验证证据已记录。
- 用户明确要求停止。**如果用户说停时仍有未解决的问题，追问一次（仅一次）：**"好的。我注意到还有 [X] 和 [Y] 未处理。需要我把它们记录到遗留问题中，还是你之后自己处理？"尊重用户的回答——用户说不用就不用，不反复追问。
- 用户改变目标，需要重新定义目的。

以下情况必须暂停、询问用户，并记录 `paused_waiting_user_input`：

- 目的或成功标准不清楚。
- 缺少必要上下文。
- 下一步操作有风险、破坏性、昂贵、外部影响或模糊。
- 多个有效方案需要用户偏好。
- 继续执行必须依赖猜测。
- 同一问题连续两次修复无效。

不要用这些说法作为关闭理由：“probably solved”“should be fine”“good enough”“the user likely means”“I will assume”。

## 防护规则

五条规则。规则 1-3 保护用户检查点之间的执行过程，规则 4-5 保护跟事记录的长期价值。

| # | 规则 | 含义 |
| --- | --- | --- |
| **1** | **不猜测** | 目的、范围、约束、用户偏好不明确时，暂停提问。永远不靠推测推进。 |
| **2** | **不跑偏** | 只做 Decide 选定的范围。不混入无关清理、重构或"顺便改一下"。新问题记入下一轮。 |
| **3** | **不跳过** | 每轮必须 Observe（读取真实当前状态，而非凭记忆）和 Verify（用具体证据验证结果）。缺了任何一个，这轮就是不完全的。 |
| **4** | **说人话** | 使用具体文件路径、行号、命令和数字。用户在检查点判断靠的是证据——含糊的报告让用户无法做知情判断。 |
| **5** | **留后路** | 将可复用模式写入轮次文件（记录模板 → Iterate 步骤）。由 `work_ooda/_index.md` 索引。 |

规则 1-3 对应执行流程：动手前不猜测，执行中不跑偏，做完后不跳过反馈循环。规则 4-5 确保每次跟事都留下可检索的痕迹——具体证据让用户今天能判断，可复用模式让明天的问题有据可查。

其他步骤级规则（Observe 读取文件、Orient 定义核心问题、Iterate 自动下一轮、两次失败暂停、Act 锁定范围、不放交付物到 work_ooda/）由各自步骤章节强制执行。

## 什么时候使用更重的质量检查

以下任务使用更深检查：

- 有测试或构建步骤的代码任务。
- 发布、部署或数据迁移。
- 安全、法律、金融或不可逆动作。
- 用户要求 review 或 audit。

普通任务保持轻量。

## 编码原则（仅编码/配置/skill 类任务）

当任务涉及代码、prompt、skill、模板或 agent 配置时，在 Verify 阶段扫一眼这些原则。**其他类型任务直接跳过本节。** 它们是设计规则，不是额外的检查清单。

**编码前先想**：假设已明确。不清楚的需求已询问。会改变结果的取舍已说明。

**简洁优先**：没有未请求功能。没有不必要抽象。选择最小可靠修复。

**外科手术式变更**：检查本轮改动是否超出 Decide 范围 → 见 Act 步骤的锁定范围约束。

**目标驱动**：→ 由 Verify 步骤的 success-standard 检查覆盖，此处不再重复。

针对 skill 包：frontmatter 使用支持的字段，引用文件存在，触发描述避免误触发，README 记录版本变更。
