# OODA

使用本 skill 运行强制 OODA 执行循环：澄清目的，观察足够的现实，围绕真实问题定向，明确决策，在范围内执行，同时验证结果和路径；未解决就自动下一轮，有价值时沉淀学习，只有在问题被验证解决后才关闭。

这是中文版。默认英文版位于 `SKILL.md`。

面向用户的输出默认使用用户所用语言，除非用户另有要求。

## 核心规则

优化目标是可靠循环，而不是完美规划或仓促执行。每个阶段保持简洁但有实质内容：输出足以证明工作仍在正确路径上的关键逻辑。

每次触发 OODA 时，都必须输出每一步的当前逻辑。说明当前步骤在做什么、为什么这样做、关键发现或决策是什么。每一步用 1-3 句话说明，不隐藏任何阶段。

发现并修复问题后，不要只跑一轮就停止。必须自动再运行一轮 OODA 来验证修复。持续迭代，直到问题解决、用户要求停止，或必须等待用户输入。

不要猜测不清楚的意图或缺失信息。如果目的、成功标准、关键上下文或风险取舍不清楚，先暂停并向用户提出一个简洁问题，再继续执行。

## 目的优先

第一轮开始前，先定义：

- 用户的真实目的。
- 本任务的成功标准。
- 已知约束、风险和必须确认的事项。

如果其中任何一项不清楚，向用户提问，不要自行推断。记录状态为 `paused_waiting_user_input`，用户回答后继续同一个 OODA 任务。

## 过程记录目录

OODA 过程文件写入 `work_ooda/`。该目录是 OODA 的工作台，用于记录思考、执行和复盘，**不是任务最终交付物目录**。

在**当前工作目录**下创建，不要创建在 skill 安装目录下：

```text
work_ooda/{YYYYMMDD-HHMMSS}-{任务名}/
```

`YYYYMMDD-HHMMSS` 是任务开始时的本地时间戳。`{任务名}` 从用户请求中提取。

使用以下文件结构：

```text
work_ooda/{YYYYMMDD-HHMMSS}-{任务名}/
├── 00-目的.md
├── 01-第一轮.md
├── 02-第二轮.md
├── ...
└── summary.md
```

每轮结束后立即写入单独文件。不要只把所有轮次写进一个 summary。

重要：`work_ooda/` 只存放 OODA 元文件：目的澄清、每轮报告、最终总结。任务实际交付物，例如代码、文档、报告或分析成果，必须写入项目约定位置，**禁止放入 `work_ooda/`**。

`00-目的.md` 必须包含：

- 用户目标：用户真正想要的效果，而不仅是任务字面表述。
- 成功标准：如何判断任务完成。
- 约束：时间、范围、安全边界、必须确认的事项。
- 预计轮次：能估计就写数字，不能估计写 `unknown`。

每个 `0{N}-第{N}轮.md` 必须包含：

- Observe 证据。
- Orient 诊断与问题分级。
- Decide 方案。
- Act 变更。
- 自检。
- 是否进入下一轮。

`summary.md` 必须包含：

- 目的回顾。
- 总轮次和最终状态：`完成`、`部分完成` 或 `需要人工`。
- 每轮摘要表。
- 成功标准验收清单。
- 仍需用户决策的问题。

记录保持简洁。不要存储密钥、凭证、私有 token 或不必要的用户数据。

## 详细输出模式

每次 OODA 都必须使用详细输出模式。以结构化方式向用户展示每一步的当前逻辑。

### 每步输出格式

| 步骤 | 输出内容 |
| --- | --- |
| Purpose | `## 目的澄清` + 用户目标、成功标准、约束条件 |
| Observe | `## Observe 观察` + 正在观察什么、当前状态、关键发现；第 2 轮起追加 Delta 对比 |
| Orient | `## Orient 定向` + 一句话根因和问题分级表 |
| Decide | `## Decide 决策` + 选择方案、选择原因、预期效果 |
| Act | `## Act 执行` + 具体操作、涉及文件、执行结果和错误信息 |
| Verify | `## Verify 验证` + 验证方式、验证结果、置信度和回归状态 |
| Iterate/Close | `## 迭代 / 关闭` + 是否进入下一轮及原因，或任务关闭总结 |

同时写入过程文件，并向用户展示步骤摘要。

## 循环规模

选择能够可靠解决任务的最小循环规模：

| 规模 | 使用场景 | 典型形态 |
| --- | --- | --- |
| 微循环 | 窄范围 bug、文档修复、对比、验证或单文件修改 | 观察相关对象，做一个范围明确的决策，执行，然后再跑一轮验证 |
| 中循环 | 功能、多文件 skill 更新、工作流设计或非平凡内容包 | 按依赖拆分，每轮处理选定问题，用测试或结构化检查验证 |
| 宏循环 | 架构、迁移、发布或高风险策略 | 尽早询问成功标准，使用可见检查点，不可逆动作前暂停确认 |

不要因为有规模分类就把普通工作变慢。规模只决定每轮需要多少证据、规划和验证。

## 最小输出契约

已完成的工作，结尾必须包含：

- 具体结果。
- 执行过的验证。
- 问题是否已解决。
- 过程记录目录：`work_ooda/{YYYYMMDD-HHMMSS}-{任务名}/`，仅包含 OODA 元文件，不是任务交付物。
- 总轮次。
- 只有当仍有下一步时才写下一步。

决策类任务必须说明：

- 选择的路径。
- 为什么该路径符合当前证据。
- 是否需要用户确认后再行动。

## 循环

### 详细输出总则

执行每一步时，都要向用户输出当前步骤的逻辑。

- 说明正在发生什么、发现了什么、为什么这样决策、执行结果是什么。
- 使用 `## 步骤名` 作为标题，下面用 1-3 句话给出具体逻辑。
- 不要只输出空标题，不要用“正在分析”这类空话，不要写冗长仪式感文字。
- 聚焦实质：展示具体观察对象、具体问题和具体决策。

### 0. Purpose（目的）

进入循环前先澄清用户目的。

输出：

```text
## 目的澄清
- 用户目标: [一句话]
- 成功标准: [列表]
- 约束条件: [列表]
- 预计轮次: [数字或 unknown]
- 状态: ready | paused_waiting_user_input
```

同时写入 `00-目的.md`。如果目的或成功标准不清楚，向用户提问并暂停。

### 1. Observe（观察）

捕获当前状态，聚焦关键信息。

必须实际读取本轮涉及的文件或对象。不要依赖记忆。

输出：

```text
## Observe 观察
- 当前状态: [描述当前实际状态]
- 关键发现: [观察到的事实、数据或问题]
```

从第 2 轮起，追加 Delta 对比：

```text
- Delta 对比:
  修复项: X -> 验证通过: Y，未生效: Z
  新引入问题: N
  遗留问题: M
```

只检查真正相关的材料：项目笔记、相关文件、最近变更、用户约束和真实目标。

### 2. Orient（定向）

快速但有证据地形成判断。

输出：

```text
## Orient 定向
- 核心问题: [一句话根因]
- 与目标的差距: [还差什么才达到成功标准]
- 问题分级:
  | 级别 | 问题 | 标签 |
  | --- | --- | --- |
  | critical/major/minor/cosmetic | ... | ... |
```

识别核心问题、根因、与目标的差距，以及有意义时的 2-3 条可行路径。倾向具体判断。证据不足就直接说明。

问题分级：

| 级别 | 含义 | 动作 |
| --- | --- | --- |
| `critical` | 阻碍目标达成 | 本轮必须修复 |
| `major` | 影响质量但不完全阻塞 | 本轮优先修复 |
| `minor` | 小瑕疵 | 成本低就修，否则记录到下一轮 |
| `cosmetic` | 格式或命名优化 | 延后或跳过 |

用问题级别决定本轮范围。

### 3. Decide（决策）

选择下一步行动。

输出：

```text
## Decide 决策
- 选择方案: [本轮要做什么]
- 选择原因: [为什么适合]
- 预期效果: [变更后的预期状态]
- 本轮范围: [只处理本轮决定的问题，按 critical -> major -> minor -> cosmetic 优先级]
```

普通低风险工作直接执行。高风险、破坏性、昂贵、外部或模糊操作，必须暂停并请求用户确认。

### 4. Act（执行）

执行选定步骤。

输出：

```text
## Act 执行
- 具体操作: [做了什么，包括命令或编辑]
- 涉及文件: [修改了哪些文件]
- 执行结果: [成功/失败；失败时写错误和处理方式]
```

执行要求：

- 使用具体命令、文件和证据。避免“可能”“应该没问题”等模糊说法。
- 锁定范围：只修 Decide 选定的问题。不要混入无关清理或重构。发现新问题记录到下一轮。
- 独立问题可并行，依赖问题串行。
- 明显错误重试一次。否则换方案或向用户求助。

### 5. Verify（验证）

检查行动是否生效。

输出：

```text
## Verify 验证
- 验证方式: [测试、文件检查、目标对比等]
- 验证结果: [问题是否解决，并给出具体证据]
- 剩余问题: [如有，列出]
- 是否引入回归: yes | no；如有标注 [regression]
- 置信度: high | medium | low
```

自检清单，写入每轮文件：

```text
[ ] 修复已生效
[ ] 无新回归，或已列出回归
[ ] 置信度: high | medium | low
[ ] 需要用户提问: yes | no
[ ] 需要下一轮: yes | no
```

验证通过前，必须同时确认：

- 结果满足用户说明的目的和成功标准。
- 执行路径没有跳过必要观察、猜测不清楚的意图，或超出已决定范围。

代码、配置、skill、prompt、模板或 agent 配置类任务，还要检查：测试或校验通过、风格一致、无未使用导入或引用、没有添加未请求功能。

### 6. Iterate（迭代）

验证发现未解决问题时，自动进入下一轮 OODA。

输出：

```text
## Iterate 迭代
- 下一轮: yes | no | paused_waiting_user_input
- 原因: [为什么需要或不需要下一轮]
```

- 未解决问题 -> 自动下一轮，并记录 `next_round: yes`。
- 受阻 -> 记录 `paused_waiting_user_input`，并提出一个简洁问题。
- 修复无效或恶化 -> 标注 `[regression]`，执行最小安全回滚，并换方案。
- 同一问题连续两轮修复无效 -> 暂停求助，记录尝试方案、回滚状态和失败原因。

只有在有长期价值时，才把可复用经验写入记忆。

### 7. Close（关闭）

只有问题被验证解决，或用户明确要求停止，才能关闭。

输出：

```text
## 任务关闭
- 最终结果: [做了什么变更]
- 验证: [验证了什么]
- 为什么已解决: [一句话]
- 过程记录: work_ooda/{YYYYMMDD-HHMMSS}-{任务名}/
- 任务交付物: [实际输出位置，非 work_ooda/]
- 总轮次: N
- 遗留: [如有]
```

关闭前写入 `summary.md`，并逐条检查 `00-目的.md` 中的成功标准。

不要仅仅因为执行了某个动作就关闭任务。只有结果和执行路径都已按用户目的验证通过，才能关闭。

## 停止与暂停规则

持续运行 OODA，直到问题被验证解决。

只有以下情况可以关闭：

- 问题已解决，并且验证证据已记录。
- 用户明确要求停止。
- 用户改变目标，需要重新定义目的。

以下情况必须暂停、询问用户，并记录 `paused_waiting_user_input`：

- 目的或成功标准不清楚。
- 缺少必要上下文。
- 下一步操作有风险、破坏性、昂贵、外部影响或模糊。
- 多个有效方案需要用户偏好。
- 继续执行必须依赖猜测。
- 同一问题连续两次修复无效。

不要用这些说法作为关闭理由：“probably solved”“should be fine”“good enough”“the user likely means”“I will assume”。

## 防护规则

用这些约束预防常见 agent 失败模式：

| 风险 | 应对 |
| --- | --- |
| 跳过观察 | 先检查最小相关上下文 |
| 没有判断就行动 | 执行前定义核心问题 |
| 假设用户批准 | 风险、破坏性、昂贵、外部或模糊操作前先问 |
| 猜测不清楚的目的 | 暂停并询问用户 |
| 汇报含糊 | 使用具体文件、命令、数量或测试结果 |
| 丢失可复用经验 | 有长期价值时写入记忆 |
| 跳过步骤输出 | 始终输出每一步的当前逻辑 |
| 首次修复后停止 | 再运行一轮验证修复 |
| Act 阶段扩大范围 | 新问题记录到下一轮 |
| 重复无效修复 | 两轮失败后暂停并询问 |
| 超出能力 | 停止并向用户求助 |
| 把交付物放进 work_ooda/ | `work_ooda/` 只放 OODA 过程记录；任务交付物放项目合适位置 |
| 优化没有问题的地方 | 不强行优化不存在问题的区域 |

## 什么时候使用更重的质量检查

以下任务使用更深检查：

- 有测试或构建步骤的代码任务。
- 发布、部署或数据迁移。
- 安全、法律、金融或不可逆动作。
- 用户要求 review 或 audit。

普通任务保持轻量。

## 代码与 Skill 包检查清单

当任务修改代码、prompt、skill、模板或 agent 配置时，在相关轮次文件中包含并向用户展示此清单：

```text
Think Before Coding:
- [ ] 假设已明确。
- [ ] 不清楚的需求已询问，而不是猜测。
- [ ] 会改变结果的取舍已说明。

Simplicity First:
- [ ] 没有添加未请求功能。
- [ ] 没有引入不必要抽象。
- [ ] 选择的是最小可靠修复。

Surgical Changes:
- [ ] 只修改本轮 Decide 范围。
- [ ] 保留既有风格和结构。
- [ ] 没有混入无关清理或重构。

Goal-Driven Execution:
- [ ] 每个改动都对应成功标准或问题。
- [ ] 验证证据已记录。
- [ ] 剩余风险或用户决策已列出。
```

针对 skill 包，还要验证：

- `SKILL.md` frontmatter 只使用目标 agent 支持的字段。
- 引用文件存在，路径相对于 skill 目录。
- 触发描述足够具体，避免误触发。
- 行为变化时，README 或 changelog 记录有意义的版本变更。
