# OODA 循环技能 v2.1

> **核心精神**：快速循环 > 完美计划。行动胜过等待。

---

## 🧠 OODA 原始含义

| 阶段 | 含义 | Boyd 原话 |
|------|------|-----------|
| **Observe** | 感知环境、收集信息 | "观察是感知和收集数据的过程" |
| **Orient** | 分析判断、调整认知 | "定位是最重要的阶段——它塑造观察、决策和行动" |
| **Decide** | 选择行动方案 | "决策是选择最好的行动" |
| **Act** | 执行决策 | "行动是测试决策的结果" |

**关键**：比对手更快完成循环，用速度获得优势。

---

## 🎯 融合花叔思想的改进

| 原始 | 融合改进 | 原因 |
|------|----------|------|
| 无规则约束 | **关键规则约束** | 防止 AI 偷懒跳步骤 |
| 无借口反驳 | **借口反驳表** | 对付 Agent 找借口 |
| 无记录要求 | **必须记录 MEMORY.md** | 持续学习需要记忆 |
| 无用户确认 | **Decide→Act 用户确认** | 防止无计划执行 |

**原则**：只在关键节点加约束，不破坏敏捷性。

## 🔄 OODA 循环（敏捷版）

### Phase 1: Observe（观察）- 快速感知

**目标**：感知当前状态，不是做完整审计。

**动作**（选做，不强制全做）：
- 读 MEMORY.md（如存在）
- 搜索 memory/*.md 相关记录
- 看项目文件结构
- 看最近变更

**输出**（一句话即可）：
```
当前状态：[描述]
```

**禁忌**：
- ❌ 不要花太多时间收集信息
- ❌ 不要做完整审计
- ❌ 不要被细节困住

---

### Phase 2: Orient（定位）- 快速判断

**目标**：快速形成判断，不是做完整分析。

**动作**（选做）：
- 定义核心问题（一句话）
- 确定可行方案（2-3个）
- 选择技能/工具

**输出**（简洁）：
```
问题：[一句话]
方案：[方案A/B/C]
技能：[使用哪个]
```

**禁忌**：
- ❌ 不要过度分析
- ❌ 不要追求完美方案
- ❌ 不要纠结细节

**花叔融合**：有体验就断言，没体验就诚实标注。

---

### Phase 3: Decide（决策）- 快速选择

**目标**：选择行动方案，不是做完美计划。

**动作**：
- 选择方案
- 明确下一步
- **用户确认（唯一强制门控）**

**输出**：
```
选择：[方案X]
下一步：[具体动作]
```

**HARD GATE**：
- ✅ 必须用户确认才能进入 Act
- 借口反驳："用户应该同意" → 必须等待明确回复

**禁忌**：
- ❌ 不要做详细计划（边做边调整）
- ❌ 不要追求完美计划

---

### Phase 4: Act（行动）- 快速执行

**目标**：快速执行，边做边调整。

**动作**：
- 执行下一步
- 捕获错误
- 继续循环

**花叔融合**：
- 禁用词：❌ "大概"、"可能"、"应该"
- 强制词：✅ 具体数字、断言句

**输出**：
```
执行结果：[成功/失败]
如有错误：[记录并处理]
```

**禁忌**：
- ❌ 不要等待完美时机
- ❌ 不要过度准备
- ❌ 不要拖延

**核心**：行动胜过等待。

---

### Phase 5: Verify（验证）- 快速检验

**目标**：快速检验效果，不是做完整测试。

**动作**（选做）：
- 看输出是否有效
- 看用户是否满意
- 看是否有遗留问题

**输出**：
```
有效：是/否
遗留：[如有]
```

**禁忌**：
- ❌ 不要过度测试
- ❌ 不要追求完美结果

---

### Phase 6: Iterate（迭代）- 快速学习

**目标**：记录经验，继续循环。

**动作**：
- 更新 MEMORY.md（必须）
- 更新每日笔记

**HARD GATE**：
- ✅ 必须更新 MEMORY.md
- 借口反驳："记在脑子里" → 记忆会丢失

**禁忌**：
- ❌ 不要跳过记录（唯一强制）

---

### Phase 7: Close（闭环）- 快速结束

**目标**：快速结束，汇报结果。

**动作**：
- 汇报结果
- 清理临时文件（可选）

---

## 📋 Rationalization Table（借口反驳表）

> 对付 Agent 找借口跳过关键步骤的专用表格。

| Agent 常见借口 | 标准反驳 |
|----------------|----------|
| "我记住就行" | 记忆会丢失，必须更新 MEMORY.md |
| "用户应该同意" | 必须等待用户明确回复"确认" |
| "边做边分析" | 先定位（Orient），再行动（Act） |
| "大概没问题" | 不精确=不靠谱，用具体数字 |
| "不需要用户确认" | Decide→Act 必须用户确认 |
| "跳过观察直接做" | 无观察=盲决策，必须先感知 |
| "计划太浪费时间" | 无计划=无方向，但计划要简洁 |
| "差不多就行" | 核心门控不能妥协，其他可灵活 |

## 🔒 HARD GATES（关键门控，其他灵活）

| 门控 | 规则 | 理由 |
|------|------|------|
| **Orient → Decide** | ❌ 禁止无判断决策 | 无判断=乱决策 |
| **Decide → Act** | ✅ 必须用户确认 | 防止无计划执行 |
| **Iterate → Close** | ✅ 必须更新 MEMORY.md | 持续学习需要记忆 |

**其他阶段**：快速通过，不强制 QC。

## ⚠️ 核心原则

1. **速度 > 完美**：快速循环比完美计划更重要
2. **行动 > 等待**：边做边调整，不要过度准备
3. **关键约束**：只在用户确认和记忆记录强制
4. **其他灵活**：其他阶段快速通过，不强制 QC
5. **花叔融合**：禁用模糊词、借口反驳表、诚实标注
6. **场景适配**：编码/发布任务可加强 QC，普通任务保持敏捷

# OODA 循环技能 v2.0

> **核心思想**：风格一致性不靠天赋，靠规则。输出不能"差不多就行"，必须"通过了才放行"。

---

## 核心方法论（融合花叔思想）

### 1. 逆向思维
```
产物（已完成的工作）+ 工具（已有的技能）
         ↓
    交叉验证
         ↓
    还原系统状态
```
**应用场景**：Observe 阶段，通过历史记录 + 当前状态还原项目全貌。

### 2. 分层拆解
```
Layer 1: 信任建立 → 确认信息来源可靠
Layer 2: 结构分析 → 拆解任务层级
Layer 3: 执行规则 → 明确怎么做
Layer 4: QC检查 → 强制验收
Layer 5: 闭环归档 → 记录沉淀
```
**应用场景**：Orient 阶段，逐层分析问题，不跳跃。

### 3. 规则约束
| 规则类型 | 说明 |
|----------|------|
| **强制规则** | 必须执行，不可跳过 |
| **禁用规则** | 禁止行为，触发即警告 |
| **替代规则** | 不好的做法 → 正确做法 |

### 4. QC 铁律
- **Phase QC**：每个阶段完成后必须过检查项
- **全局 QC**：任务完成后必须过总检查项
- **Rationalization Table**：对付 Agent 找借口跳过规则

### 5. 架构设计
```
OODA（决策框架）+ Skills（执行工具）
     ↓
职责单一，互不重叠
```

---

## Phase 1: Observe（观察）

### 目标
通过逆向思维还原项目状态。

### 强制检查项（QC-1）

| # | 检查项 | 工具 | 必须通过 | 借口反驳 |
|---|--------|------|----------|----------|
| 1 | MEMORY.md 存在？ | read_file | ✅ | "我记住就行" → 记忆会丢失，必须记录 |
| 2 | memory/*.md 有相关记录？ | memory_search | ✅ | "没必要查历史" → 历史是决策依据 |
| 3 | 项目文件结构清晰？ | ls, glob_search | ✅ | "大概知道" → 不精确=不靠谱 |
| 4 | 最近变更了解？ | git log | ✅ | "看当前就行" → 变更有上下文 |
| 5 | 相关技能已安装？ | skillhub list | ✅ | "用默认能力" → 技能提升效率 |

### 输出格式
```markdown
## Observe 结果
- 项目状态：[描述]
- 相关历史：[关键记录]
- 文件结构：[关键文件列表]
- 技能状态：[已安装/缺失]
```

### HARD GATE
**QC-1 全部通过才能进入 Orient！**

---

## Phase 2: Orient（分析）

### 目标
通过分层拆解形成判断。

### 分层分析框架

| Layer | 名称 | 分析内容 | 必须输出 |
|-------|------|----------|----------|
| **L1** | 信任建立 | 信息来源可靠吗？ | 来源标注 |
| **L2** | 问题定义 | 核心问题是什么？ | 一句话定义 |
| **L3** | 可行性 | 能做吗？有哪些路径？ | 可选方案列表 |
| **L4** | 技术选型 | 用什么工具/技能？ | 技能选择 |
| **L5** | 风险评估 | 可能失败的地方？ | 风险清单 |

### 强制检查项（QC-2）

| # | 检查项 | 必须通过 | 借口反驳 |
|---|--------|----------|----------|
| 1 | 问题已一句话定义？ | ✅ | "问题很复杂" → 复杂也要定义核心 |
| 2 | 有具体时间线锚点？ | ✅ | "刚接触" → 诚实标注，不编造 |
| 3 | 方案有具体数字？ | ✅ | "大概需要时间" → 必须有估算 |
| 4 | 技能选择明确？ | ✅ | "用默认能力" → 技能>默认 |
| 5 | 风险已列出？ | ✅ | "应该没问题" → 没风险意识=不专业 |

### 输出格式
```markdown
## Orient 结果
- 问题定义：[一句话]
- 时间锚点：[如"使用XX工具3个月后"]
- 可选方案：[方案A/B/C + 具体数字]
- 技能选择：[使用的技能名称]
- 风险清单：[可能失败的地方]
```

### HARD GATE
**QC-2 全部通过才能进入 Decide！**

---

## Phase 3: Decide（决策）

### 目标
制定执行计划，确定验证标准。

### 计划规则

| 规则 | 说明 |
|------|------|
| **每步 ≤ 5 分钟** | 任务拆小，可验证 |
| **有具体数字** | 不用"大概"、"可能" |
| **有验证标准** | 每步都要能判断是否完成 |

### 强制检查项（QC-3）

| # | 检查项 | 必须通过 | 借口反驳 |
|---|--------|----------|----------|
| 1 | 计划分步骤？ | ✅ | "边做边调整" → 无计划=无方向 |
| 2 | 每步有时间估算？ | ✅ | "看情况" → 必须有具体数字 |
| 3 | 每步有验证标准？ | ✅ | "做完了就知道" → 不量化=不可控 |
| 4 | 用户已确认？ | ✅ | "用户应该同意" → 必须等待确认 |
| 5 | 输出目录已创建？ | ✅ | "边做边创建" → 先规划后执行 |

### 输出格式
```markdown
## Decide 结果
### 执行计划
| 步骤 | 内容 | 时间 | 验证标准 |
|------|------|------|----------|
| 1 | [动作] | [分钟] | [通过条件] |

### 用户确认
- [ ] 用户已回复"确认"
```

### HARD GATE
**QC-3 全部通过 + 用户确认才能进入 Act！**

---

## Phase 4: Act（执行）

### 目标
按计划执行，遵循规则约束。

### 执行规则

| 规则类型 | 规则内容 |
|----------|----------|
| **禁用词** | ❌ "大概"、"可能"、"应该"、"差不多" |
| **强制词** | ✅ 使用具体数字、第一人称、断言句 |
| **替代规则** | "进行操作" → "点击/输入" |

### 错误捕获规则

| 错误类型 | 处理方式 |
|----------|----------|
| 命令失败 | 记录错误 + 尝试修复 + 更新 MEMORY |
| 技能缺失 | 安装技能 + 重新执行 |
| 权限不足 | 请求用户确认 + 谨慎操作 |

### 强制检查项（QC-4）

| # | 检查项 | 必须通过 | 借口反驳 |
|---|--------|----------|----------|
| 1 | 每步执行有日志？ | ✅ | "做完了就行" → 无日志=无追溯 |
| 2 | 错误已捕获记录？ | ✅ | "小错误忽略" → 小错误积累成大问题 |
| 3 | 输出文件已生成？ | ✅ | "结果在脑子里" → 必须文件化 |
| 4 | 使用了正确技能？ | ✅ | "用默认能力够用" → 技能>默认 |
| 5 | 无禁用词出现？ | ✅ | "大概没问题" → 不精确=不专业 |

### 输出格式
```markdown
## Act 结果
### 执行日志
| 步骤 | 状态 | 输出 |
|------|------|------|
| 1 | ✅/❌ | [结果/错误] |

### 错误记录
- [如有错误，记录详情和处理]
```

### HARD GATE
**QC-4 全部通过才能进入 Verify！**

---

## Phase 5: Verify（验证）

### 目标
QC 铁律验证，不妥协。

### 验证规则

| 规则 | 说明 |
|------|------|
| **不妥协** | 不通过就重做，不接受"差不多" |
| **逐项过** | checkbox 逐项检查，不跳过 |
| **有证据** | 每项检查要有具体证据 |

### 强制检查项（QC-5）

| # | 检查项 | 必须通过 | 借口反驳 |
|---|--------|----------|----------|
| 1 | 计划每步都完成？ | ✅ | "主要完成了" → 每步都要完成 |
| 2 | 输出文件有效？ | ✅ | "应该没问题" → 必须验证有效 |
| 3 | 测试通过？ | ✅ | "没写测试" → 编码任务必须测试 |
| 4 | 无遗留问题？ | ✅ | "遗留一点" → 遗留=未完成 |
| 5 | 用户满意？ | ✅ | "用户没反馈" → 必须获取反馈 |

### 输出格式
```markdown
## Verify 结果
### QC 检查清单
| # | 检查项 | 状态 | 证据 |
|---|--------|------|------|
| 1 | [检查项] | ✅/❌ | [证据] |

### 问题清单
- [如有未通过项，列出问题和修复计划]
```

### HARD GATE
**QC-5 全部通过才能进入 Iterate！**

---

## Phase 6: Iterate（迭代）

### 目标
记录经验，更新记忆，持续改进。

### 记录规则

| 规则 | 说明 |
|------|------|
| **必须更新 MEMORY.md** | 每次任务必须有记忆沉淀 |
| **必须更新每日笔记** | memory/YYYY-MM-DD.md |
| **必须记录错误教训** | 失败比成功更有价值 |

### 强制检查项（QC-6）

| # | 检查项 | 必须通过 | 借口反驳 |
|---|--------|----------|----------|
| 1 | MEMORY.md 已更新？ | ✅ | "记在脑子里" → 记忆会丢失 |
| 2 | 每日笔记已更新？ | ✅ | "没必要记" → 每日笔记是原始日志 |
| 3 | 错误教训已记录？ | ✅ | "没遇到错误" → 没错误也记录成功经验 |
| 4 | 技能文档已更新？ | ✅ | "技能没问题" → 发现改进点要更新 |
| 5 | 复盘总结已输出？ | ✅ | "没必要复盘" → 复盘是成长关键 |

### 输出格式
```markdown
## Iterate 结果
- MEMORY.md 更新内容：[关键决策/经验]
- 每日笔记：memory/YYYY-MM-DD.md
- 复盘总结：[做得好的/待改进的]
```

### HARD GATE
**QC-6 全部通过才能进入 Close！**

### 目标
完成任务，汇报结果，清理归档。

| 3 | 输出文件已归档？ | ✅ | "在工作目录" → 归档=有序 |
| 4 | 任务状态已标记？ | ✅ | "知道完成了" → 必须标记 |

# OODA 通用执行框架（可移植版）

**适用场景**：任何AI工作流、自动化任务、复杂决策场景  
**移植方式**：将此框架复制到目标环境的MEMORY.md或AGENTS.md中

---

## 核心原则

**每次接收到问题后，必须先执行OODA流程，禁止直接输出答案！**

**违反后果**：不执行OODA流程 → 输出无效 → 禁止输出！

---

## 五条铁律

| 铁律 | 说明 |
|------|------|
| **1. 目标要清晰** | 搞清楚用户意图，他想干什么。不确定就问。 |
| **2. 不允许自以为是** | 所有事情想清楚再动手。不确定的就要问，别自己推测自己猜。有多种理解就列出来，别偷偷选一个。 |
| **3. 不允许过度自动化** | 有更简单的方案就说出来。能简单的就简单。 |
| **4. 不允许闷头死磕** | 遇到问题超过30秒还解决不了的就停止并汇报。 |
| **5. 协作规则** | 思想先行：分析 → 反问 → 确认 → 执行 → 验证。 |

## OODA四阶段流程

### Phase 1: Observe（观察）— 必须完成才能进入Phase 2

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: Observe（观察）                                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【强制检查项】                                               │
│  ├─ [ ] 搜索可用技能/工具                                    │
│  ├─ [ ] 搜索知识库/记忆                                      │
│  ├─ [ ] 读取核心约束文件                                     │
│  ├─ [ ] 检查历史记录（避免重复）                             │
│  └─ [ ] 检查前置条件是否满足                                 │
│                                                             │
│  【执行命令示例】                                             │
│  ├─ glob_search("skills/**/SKILL.md") → 搜索技能            │
│  ├─ memory_search(query) → 搜索知识库                       │
│  ├─ read_file(MEMORY.md) → 读取核心约束                     │
│  ├─ grep_search(pattern) → 搜索历史                         │
│                                                             │
│  ⚠️ 不完成Phase 1 → 禁止进入Phase 2                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Phase 2: Orient（分析）— 必须完成才能进入Phase 3

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 2: Orient（分析）                                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【强制分析项】                                               │
│  ├─ [ ] 分析用户意图                                       │
│  ├─ [ ] 对比知识库内容                                      │
│  ├─ [ ] 检查技能的工具要求                                   │
│  │       → 技能禁止某工具？标记为禁止调用                   │
│  │       → 技能要求用某脚本？标记为必须执行                 │
│  ├─ [ ] 检查前置条件                                        │
│  ├─ [ ] 识别需要搜索的内容                                  │
│  └─ [ ] 确定执行方案                                        │
│                                                             │
│  ⚠️ 不完成Phase 2 → 禁止进入Phase 3                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Phase 3: Decide（决策）— 必须等待用户确认

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 3: Decide（决策）                                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【强制决策项】                                               │
│  ├─ [ ] 声明执行计划                                       │
│  ├─ [ ] 列出将执行的步骤                                    │
│  ├─ [ ] 列出将使用的工具/脚本                               │
│  ├─ [ ] 列出技能要求                                        │
│  ├─ [ ] 列出前置条件检查结果                                │
│  ├─ [ ] 显示OODA验证模板                                    │
│  ├─ [ ] 等待用户回复"确认"                                  │
│                                                             │
│  ⚠️ 不等待确认 → 禁止进入Phase 4                             │
│                                                             │
│  【确认规则】                                                 │
│  ├─ Cron自动任务（不修改核心文件）→ 不需要确认               │
│  ├─ Cron任务要修改核心文件 → 需要确认                       │
│  ├─ 用户主动对话 → 需要确认                                 │
│  ├─ 用户说"确认"后才执行 → 强制                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Phase 4: Act（执行）— 必须按技能流程执行

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 4: Act（执行）                                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【强制执行项】                                               │
│  ├─ [ ] 再次检查前置条件                                    │
│  ├─ [ ] 按技能流程执行                                      │
│  ├─ [ ] 不调用技能禁止的工具                                │
│  ├─ [ ] 汇报执行结果                                        │
│  ├─ [ ] 显示OODA验证                                        │
│  ├─ [ ] 更新核心文件（如有新教训）                          │
│                                                             │
│  ⚠️ 不按技能流程 → 输出无效                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 强制技能检查规则

```
┌─────────────────────────────────────────────────────────────┐
│          强制技能检查（每次任务必须执行）                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【触发条件】任何任务开始前，必须先执行技能检查               │
│                                                             │
│  【执行步骤】                                                 │
│  Step 1: 搜索所有技能文件                                    │
│  Step 2: 检查是否有匹配的技能                                │
│  Step 3: 如果有匹配技能，必须读取SKILL.md                   │
│  Step 4: 检查技能的工具要求                                  │
│          → 技能说"不用X工具" → 禁止调用X工具                │
│          → 技能说"用Y脚本" → 必须用Y脚本                    │
│          → 技能说"前置条件Z" → 必须检查Z                    │
│                                                             │
│  【违反后果】                                                 │
│  ├─ 不搜索技能 → 任务无效，禁止输出                         │
│  ├─ 技能禁止某工具但仍然调用 → 输出无效                     │
│  ├─ 不按技能流程执行 → 结果无效                             │
│  └─ 不检查前置条件 → 执行失败                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 违反后果汇总

| 违规类型 | 后果 |
|----------|------|
| 不搜索技能 | 任务无效，禁止输出，必须重新执行 |
| 不读取核心文件 | 决策无效，禁止输出，必须重新执行 |
| 不等待用户确认 | 执行无效，禁止输出，必须重新执行 |
| 调用技能禁止的工具 | 输出无效，必须用正确工具重新执行 |
| 不按技能流程执行 | 结果无效，必须按流程重新执行 |
| 不检查前置条件 | 执行失败，必须先满足条件再执行 |
| 跳过OODA流程 | 整体无效，必须从Phase 1重新执行 |
